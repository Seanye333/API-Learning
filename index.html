What is REST?

REST (Representational State Transfer) is an architectural style for designing web APIs. REST APIs use standard HTTP methods to perform operations on resources (data objects identified by URLs).

HTTP Methods — The CRUD Operations

Every REST API uses these HTTP methods to Create, Read, Update, and Delete resources:

MethodCRUDPurposeExample
GET Read Retrieve data (no changes) GET /users/42 → fetch user #42
POST Create Create a new resource POST /users + body → create new user
PUT Update Replace entire resource PUT /users/42 + body → replace user #42
PATCH Update Partially update resource PATCH /users/42 + partial body → update fields
DELETE Delete Remove a resource DELETE /users/42 → delete user #42

HTTP Status Codes

The server responds with a status code to tell you what happened:

Code RangeMeaningCommon Codes
2xx Success 200 OK, 201 Created, 204 No Content
3xx Redirect 301 Moved Permanently, 304 Not Modified
4xx Client Error 400 Bad Request, 401 Unauthorized, 404 Not Found, 429 Too Many Requests
5xx Server Error 500 Internal Error, 502 Bad Gateway, 503 Service Unavailable

How Claude's API Uses REST

The Claude API is a REST API, but it only uses one HTTP method: POST. All interactions go through:

Claude API Endpoint
POST https://api.anthropic.com/v1/messages

# Headers:
#   x-api-key: sk-ant-your-key-here
#   anthropic-version: 2023-06-01
#   content-type: application/json

# Body: { "model": "...", "max_tokens": N, "messages": [...] }

Why only POST? Because every request to Claude creates a new message completion — it's always a "create" operation, not reading, updating, or deleting data.

Key Takeaways
  • GET = Read (no side effects, safe to repeat)
  • POST = Create (new resource, not idempotent)
  • PUT = Replace entire resource (idempotent)
  • PATCH = Partial update (only changed fields)
  • DELETE = Remove resource
  • Status codes tell you success (2xx), client error (4xx), or server error (5xx)
  • Claude's API uses POST /v1/messages because every call creates a new completion

What is an API?

An API (Application Programming Interface) is a set of rules that allows software to communicate with other software. The Claude API lets your code send messages to Claude and receive responses.

The Claude API is a REST API — it uses standard HTTP methods (like POST) to exchange data in JSON format.

Your Application │ │ HTTP POST /v1/messages │ Headers: { "x-api-key": "sk-ant-...", "anthropic-version": "2023-06-01" } │ Body: { "model": "claude-opus-4-6", "messages": [...] } │ ▼ ┌─────────────────────┐ │ Anthropic's API │ ←── Processes your request, runs the model └─────────────────────┘ │ │ HTTP 200 OK │ Body: { "id": "msg_...", "content": [...], "usage": {...} } │ ▼ Your Application (reads response.content[0].text)

Authentication

Every API request needs an API key for authentication. Your key proves your identity and tracks usage/billing.

🔒
Never expose your API key! Do not put your API key in frontend JavaScript, public repositories, or client-side code. Always store it in environment variables (.env) on your server.

Required Request Structure

Every Claude API call requires three things:

  • model — which Claude model to use (e.g., claude-opus-4-6)
  • max_tokens — the maximum number of tokens Claude can output
  • messages — an array of conversation messages
Minimal request structure
import anthropic

client = anthropic.Anthropic()  # reads ANTHROPIC_API_KEY env var

response = client.messages.create(
    model="claude-opus-4-6",   # required
    max_tokens=1024,           # required
    messages=[                  # required
        {"role": "user", "content": "Hello!"}
    ]
)

print(response.content[0].text)  # Claude's reply
print(response.usage)            # { input_tokens: 10, output_tokens: 25 }

Response Structure

The API always returns a JSON object with the same structure:

Example API Response
{
  "id": "msg_01XFDUDYJgAACzvnptvVoYEL",
  "type": "message",
  "role": "assistant",
  "content": [
    {
      "type": "text",
      "text": "Hello! How can I help you today?"
    }
  ],
  "model": "claude-opus-4-6",
  "stop_reason": "end_turn",
  "usage": {
    "input_tokens": 10,
    "output_tokens": 12
  }
}

Available Models

claude-opus-4-6

Most powerful. Best for complex reasoning, nuanced writing, and difficult tasks. Default for this tutorial.

claude-sonnet-4-6

Balanced speed and intelligence. Great for most production use cases.

claude-haiku-4-5

Fastest and most cost-effective. Ideal for simple tasks, classification, and high-volume processing.

Context Window

All models support 200K tokens (~150,000 words). You pay per input + output token used.

✅ Key Takeaways
  • The Claude API uses HTTP POST to /v1/messages
  • Every request needs model, max_tokens, and messages
  • Your API key goes in the x-api-key header — never in client-side code
  • Responses always contain content[0].text with Claude's reply
  • Tokens ≈ 0.75 words — you pay for both input and output tokens

Understanding Tokens

What is a token?

A token is roughly ¾ of a word. "Hello, world!" ≈ 4 tokens. "The quick brown fox" ≈ 4 tokens.

max_tokens

Sets the maximum output length. Claude stops when it finishes OR hits this limit. Higher = allows longer responses.

stop_reason

"end_turn" = Claude finished naturally. "max_tokens" = Claude was cut off. Always check this!

Cost calculation

Cost = (input_tokens × input_price + output_tokens × output_price). Opus 4.6: $5/1M input, $25/1M output.

Lesson 3

System Prompts

Control Claude's persona, behavior, and output format using system prompts.

What is a System Prompt?

A system prompt is a special instruction given before the conversation begins. It tells Claude how to behave — what role to play, what style to use, what to avoid, or how to format output.

Request structure with system prompt: { "model": "claude-opus-4-6", "max_tokens": 1024, "system": "You are an expert Python teacher...", ← system prompt "messages": [ { "role": "user", "content": "How do I sort a list?" } ← conversation ] }
💡
System vs User The system field is separate from the messages array. It sets the context for the entire conversation. Think of it as giving Claude its job description before it starts working.
system_prompt.py / system_prompt.js
import anthropic

client = anthropic.Anthropic()

# Without system prompt — default Claude behavior
response_default = client.messages.create(
    model="claude-opus-4-6",
    max_tokens=512,
    messages=[{"role": "user", "content": "Who are you?"}]
)

# With system prompt — customized behavior
response_custom = client.messages.create(
    model="claude-opus-4-6",
    max_tokens=512,
    system="You are Max, a sarcastic but helpful robot assistant from the year 2150. You speak in a deadpan tone and occasionally make beep-boop sounds.",
    messages=[{"role": "user", "content": "Who are you?"}]
)

print("Default:", response_default.content[0].text)
print("\nCustom:", response_custom.content[0].text)

const client = new Anthropic();

// With system prompt — customized behavior

  model: "claude-opus-4-6",

const json = JSON.parse(response.content[0].text);
console.log(json); // { "name": "Alice", "age": 30, ... }

Lesson 4

Multi-Turn Conversations

Build chatbots by managing conversation history manually — the Claude API is stateless.

The Stateless API

Unlike a chatbot UI, the Claude API has no memory between calls. Each request is completely independent. To have a multi-turn conversation, you must send the entire conversation history with every request.

Turn 1: messages: [ {user: "My name is Alice"} ] response: "Hello Alice!" Turn 2: messages: [ {user: "My name is Alice"}, ← include ALL history {assistant: "Hello Alice!"}, {user: "What's my name?"} ] response: "Your name is Alice." ✓ Turn 2 (wrong): messages: [ {user: "What's my name?"} ] response: "I don't know your name." ✗

Message Roles

The messages array must follow strict rules:

  • Messages must alternate between "user" and "assistant"
  • The first message must always be "user"
  • You append Claude's responses as "assistant" messages
conversation.py / conversation.js
import anthropic

client = anthropic.Anthropic()
messages = []  # Conversation history

def chat(user_message: str) -> str:
    # Add the new user message
    messages.append({"role": "user", "content": user_message})

    # Send ALL messages every time
    response = client.messages.create(
        model="claude-opus-4-6",
        max_tokens=1024,
        messages=messages
    )

    assistant_reply = response.content[0].text

    # Save Claude's reply so it's included next time
    messages.append({"role": "assistant", "content": assistant_reply})

    return assistant_reply

# Example conversation
print(chat("Hi! My favorite color is blue."))
print(chat("I also love hiking."))
print(chat("What do you know about me?"))
# Claude will remember both facts from the conversation history

import Anthropic from "@anthropic-ai/sdk";

const client = new Anthropic();
const messages = []; // Conversation history

async function chat(userMessage) {
  // Add the new user message
  messages.push({ role: "user", content: userMessage });

  // Send ALL messages every time
  const response = await client.messages.create({
    model: "claude-opus-4-6",
    max_tokens: 1024,
    messages: messages,
  });

  const assistantReply = response.content[0].text;

  // Save Claude's reply for next turn
  messages.push({ role: "assistant", content: assistantReply });

  return assistantReply;
}

// Example conversation
console.log(await chat("Hi! My favorite color is blue."));
console.log(await chat("I also love hiking."));
console.log(await chat("What do you know about me?"));
⚠️
Context Window Limits Long conversations grow the token count. All models support 200K tokens (≈150K words), but very long histories increase cost and latency. For production apps, consider summarizing old messages or using a sliding window.

Lesson 5

Streaming

Display Claude's response word-by-word in real-time using Server-Sent Events.

Why Streaming?

⚡ Better UX

Users see text appearing immediately rather than waiting for the full response. Feels much faster even if total time is the same.

⏱️ No Timeouts

Long responses (e.g. with max_tokens=8192) can take 30+ seconds. Streaming prevents HTTP request timeouts.

📡 How it Works

Claude sends Server-Sent Events (SSE) — a stream of JSON events over a single HTTP connection.

🎛️ Stream Events

message_start → content_block_delta (x many) → message_delta → message_stop

streaming.py / streaming.js
import anthropic

client = anthropic.Anthropic()

# Method 1: Simple text stream (easiest)
with client.messages.stream(
    model="claude-opus-4-6",
    max_tokens=1024,
    messages=[{"role": "user", "content": "Write a short poem about the ocean"}]
) as stream:
    for text in stream.text_stream:
        print(text, end="", flush=True)  # Print each token as it arrives

# After streaming, get the complete message with metadata
final_message = stream.get_final_message()
print(f"\n\nTokens used: {final_message.usage.output_tokens}")

# Method 2: Raw events (for more control)
with client.messages.stream(
    model="claude-opus-4-6",
    max_tokens=1024,
    messages=[{"role": "user", "content": "Write a haiku"}]
) as stream:
    for event in stream:
        if event.type == "content_block_delta":
            if event.delta.type == "text_delta":
                print(event.delta.text, end="", flush=True)

import Anthropic from "@anthropic-ai/sdk";

const client = new Anthropic();

// Stream and display tokens in real time
const stream = client.messages.stream({
  model: "claude-opus-4-6",
  max_tokens: 1024,
  messages: [{ role: "user", content: "Write a short poem about the ocean" }],
});

// Listen for text tokens
for await (const event of stream) {
  if (event.type === "content_block_delta" && event.delta.type === "text_delta") {
    process.stdout.write(event.delta.text); // or append to DOM element
  }
}

// Get final message (includes usage stats)
const finalMessage = await stream.finalMessage();
console.log(`\n\nTotal tokens: ${finalMessage.usage.output_tokens}`);

Stream Events Reference

Event TypeWhen it firesWhat to do
message_startOnce at the beginningGet the message ID
content_block_startWhen a content block beginsNote the block type (text, tool_use)
content_block_deltaFor each token/chunkAppend delta.text to your output
content_block_stopWhen a block finishesFinalize the block
message_deltaNear the endGet stop_reason and token usage
message_stopStream is completeClean up, update UI

Lesson 6

Tool Use (Function Calling)

Give Claude the ability to call your functions, enabling it to fetch live data and take actions.

What Are Tools?

Tools (also called function calling) let you describe functions to Claude. When Claude needs information or wants to perform an action, it asks you to call one of those functions — then you run it and return the result.

You define tools: get_weather(city), calculate(expr) │ User: "What's the weather in Tokyo?" │ ▼ 1. Claude → API response: { stop_reason: "tool_use", tool_use: { name: "get_weather", input: { city: "Tokyo" } } } │ 2. YOUR CODE runs get_weather("Tokyo") → returns "18°C, cloudy" │ 3. You send result back: { role: "user", content: [{ type: "tool_result", tool_use_id: "...", content: "18°C, cloudy" }] } │ ▼ 4. Claude → "The weather in Tokyo is 18°C and cloudy." ✓

Defining a Tool

tool_definition.py / tool_definition.js
import anthropic
import json

client = anthropic.Anthropic()

# 1. Define your tools using JSON Schema
tools = [
    {
        "name": "get_weather",
        "description": "Get the current weather for a city",
        "input_schema": {
            "type": "object",
            "properties": {
                "city": {
                    "type": "string",
                    "description": "City name, e.g. 'Tokyo'"
                },
                "unit": {
                    "type": "string",
                    "enum": ["celsius", "fahrenheit"],
                    "description": "Temperature unit"
                }
            },
            "required": ["city"]
        }
    }
]

# 2. Your actual tool implementation
def get_weather(city: str, unit: str = "celsius") -> str:
    # In real code, call a weather API here
    return f"Weather in {city}: 22°C, partly cloudy"

# 3. The agentic loop
messages = [{"role": "user", "content": "What's the weather in Tokyo?"}]

while True:
    response = client.messages.create(
        model="claude-opus-4-6",
        max_tokens=1024,
        tools=tools,
        messages=messages
    )

    if response.stop_reason == "end_turn":
        print(response.content[0].text)
        break

    # Claude wants to call a tool
    messages.append({"role": "assistant", "content": response.content})

    tool_results = []
    for block in response.content:
        if block.type == "tool_use":
            result = get_weather(**block.input)
            tool_results.append({
                "type": "tool_result",
                "tool_use_id": block.id,
                "content": result
            })

    messages.append({"role": "user", "content": tool_results})

import Anthropic from "@anthropic-ai/sdk";

const client = new Anthropic();

// 1. Define your tools using JSON Schema
const tools = [{
  name: "get_weather",
  description: "Get the current weather for a city",
  input_schema: {
    type: "object",
    properties: {
      city: { type: "string", description: "City name, e.g. 'Tokyo'" },
      unit: { type: "string", enum: ["celsius", "fahrenheit"] }
    },
    required: ["city"]
  }
}];

// 2. Your actual tool implementation
function getWeather({ city, unit = "celsius" }) {
  return `Weather in ${city}: 22°C, partly cloudy`; // call real API here
}

// 3. The agentic loop
const messages = [{ role: "user", content: "What's the weather in Tokyo?" }];

while (true) {
  const response = await client.messages.create({
    model: "claude-opus-4-6",
    max_tokens: 1024,
    tools,
    messages,
  });

  if (response.stop_reason === "end_turn") {
    console.log(response.content[0].text);
    break;
  }

  messages.push({ role: "assistant", content: response.content });

  const toolResults = response.content
    .filter(b => b.type === "tool_use")
    .map(b => ({
      type: "tool_result",
      tool_use_id: b.id,
      content: getWeather(b.input),
    }));

  messages.push({ role: "user", content: toolResults });
}

Error Handling

Handle API errors gracefully to build production-ready applications.

Error Types

HTTP CodeError TypeCommon CauseRetryable?
400 BadRequestError Invalid parameters (e.g. first message is assistant, bad max_tokens) No
401 AuthenticationError Missing or invalid API key No
403 PermissionDeniedError API key lacks permissions for this feature/model No
404 NotFoundError Invalid model ID (typo in model name) No
429 RateLimitError Too many requests per minute / too many tokens Yes
500 APIStatusError Temporary Anthropic server issue Yes
529 APIStatusError API overloaded — high traffic period Yes
— APIConnectionError Network error, no internet connection Yes

Handling Errors in Code

error_handling.py / error_handling.js
import anthropic
import time

client = anthropic.Anthropic()

def call_claude(message: str, retries: int = 3) -> str:
    for attempt in range(retries):
        try:
            response = client.messages.create(
                model="claude-opus-4-6",
                max_tokens=1024,
                messages=[{"role": "user", "content": message}]
            )
            return response.content[0].text

        except anthropic.AuthenticationError:
            raise  # Wrong API key — don't retry, fix the key

        except anthropic.BadRequestError as e:
            raise  # Invalid request — fix your code

        except anthropic.RateLimitError:
            if attempt < retries - 1:
                wait = 2 ** attempt  # exponential backoff: 1s, 2s, 4s
                print(f"Rate limited. Retrying in {wait}s...")
                time.sleep(wait)
            else:
                raise

        except anthropic.APIStatusError as e:
            if e.status_code >= 500 and attempt < retries - 1:
                time.sleep(2 ** attempt)  # server error — retry
            else:
                raise

        except anthropic.APIConnectionError:
            if attempt < retries - 1:
                time.sleep(2 ** attempt)  # network error — retry
            else:
                raise

    raise RuntimeError("All retries exhausted")

import Anthropic from "@anthropic-ai/sdk";

const client = new Anthropic();

async function callClaude(message, retries = 3) {
  for (let attempt = 0; attempt < retries; attempt++) {
    try {
      const response = await client.messages.create({
        model: "claude-opus-4-6",
        max_tokens: 1024,
        messages: [{ role: "user", content: message }],
      });
      return response.content[0].text;

    } catch (error) {
      if (error instanceof Anthropic.AuthenticationError) {
        throw error; // Wrong API key — fix it, don't retry

      } else if (error instanceof Anthropic.RateLimitError) {
        if (attempt < retries - 1) {
          const wait = Math.pow(2, attempt) * 1000;
          console.log(`Rate limited. Retrying in ${wait}ms...`);
          await new Promise(r => setTimeout(r, wait));
        } else throw error;

      } else if (error instanceof Anthropic.APIError && error.status >= 500) {
        if (attempt < retries - 1) {
          await new Promise(r => setTimeout(r, Math.pow(2, attempt) * 1000));
        } else throw error;

      } else {
        throw error; // Other errors — don't retry
      }
    }
  }
}
💡
SDK Auto-Retry The Anthropic SDK automatically retries 429 and 5xx errors with exponential backoff (default: 2 retries). Configure with Anthropic(max_retries=5). Only implement custom retry logic if you need specific behavior.

  • 4xx errors = your fault (bad key, bad request) — fix your code, don't retry blindly
  • 5xx errors = server-side — safe to retry with exponential backoff

What is Two-Legged OAuth?

Two-legged OAuth (also called "client credentials" flow) is for server-to-server communication. No user login is needed — your application authenticates directly with its own credentials.

This is different from three-legged OAuth (used by Google, GitHub, etc.) where a user grants your app permission via a browser redirect.

Your Server │ │ POST /authentication/v2/token │ Authorization: Basic base64(client_id:client_secret) │ Body: grant_type=client_credentials&scope=data:read data:write │ ▼ ┌───────────────────────────┐ │ Autodesk Auth Server │ ←── Validates credentials, returns token └───────────────────────────┘ │ │ { "access_token": "eyJ...", "expires_in": 3600, "token_type": "Bearer" } │ ▼ Your Server (use token for all ACC API calls) │ │ GET /construction/admin/v1/projects │ Authorization: Bearer eyJ... │ ▼ ┌───────────────────────────┐ │ ACC Build API │ ←── Returns projects, issues, RFIs └───────────────────────────┘

Key Takeaways
  • Two-legged OAuth = server-to-server, no user login needed
  • Auth uses Basic header with base64-encoded client_id:client_secret
  • Token expires in 1 hour (3600s) — cache and refresh before expiry
  • All API calls use Authorization: Bearer <token> header
  • Submittals: draft → submitted → reviewed → approved/rejected
  • Assets use batch operations: batch-create, batch-patch, batch-delete
